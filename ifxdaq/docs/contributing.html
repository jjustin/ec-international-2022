How To Contribute

First off, thank you for considering contributing to ifxdaq!

This document intends to make contribution more accessible by codifying tribal knowledge and expectations. Don’t be afraid to open half-finished PRs, and ask questions if something is unclear!

Local Development Environment

Python

We use Python to develop our package. The main development version is Python3.9 (supported versions: 3.7-3.9). Please install this version according to your OS.

Clone repository

First, you need a copy of the code. The code base is hosted in a Bitbucket repository. We use git to version control our code. You can clone the repository with:

git clone ssh://git@bitbucket.vih.infineon.com:7999/sensys/ifxdaq.git
# Change into the cloned repository
cd ifxdaq

Virtual environment

Next, we create a virtual environment. Virtual environments are a way to handle packages and dependencies for different projects on the same machine. It’s out of scope for this document to list all the ways to manage virtual environments in Python, but you’ll find a lot of tutorials in the web. We recommend using Python’s built-in venv to manage your virtual environment. We create now a virtual environment and activate it.

# Inside ifxdaq
python -m venv .venv
# Activate the environment on Windows (for other OS have a look into the docs)
.venv\Scripts\activate

Every time you start a new console and want to use the virtual environment, you have to activate it.

VSCode - virtual environment integration

PyCharm - virtual environment integration

Install dependencies

Now, we have a local copy of the repository and an activated virtual environment. We install the development environment and all dependencies with:

pip install -e .[dev]
# Add the AI profile if you want to use / work on the AI algorithms
pip install -e .[dev,ai]

This resolves the dependencies for used packages and tools automatically.

pre-commit

We use pre-commit to check commits before they are published. The available hooks will check your files for correct formatting, syntax errors and more. Please install the git hook scripts by running:

pre-commit install

Style guide

A good orientation for a style guide is the Google Style Guide. If you stick roughly to those guidelines, your code quality is probably already quite nice!

Here we want to point out a few important topics for this project that are already contained in the same manner or very similar in Google’s style guide.

Code style

We use black and isort to format our code automatically, so there is no need to pay special attention to this topic. (Note: Integrating those tools into your IDE might be really helpful.)

Documentation

Contributed code should be documented already during creation. This simplifies cooperation between multiple developers and also re-use in other projects. We follow the Google Docstring style to document our code inline. Here you find an extensive example of this style.

You can also add explicit documentation under docs. Just have a look into this folder to find examples how this is done.

Testing

Try to add tests for your code. This will ensure high quality and simplifies the development, when the complexity of the project increases. Tests are added under tests. Have a look into already existing tests to get an impression, how this can be done.

Typing

You are strongly encouraged to enable Python type analysis when updating code. Type annotations improve the readability and maintainability of your code. The type checker will convert many runtime errors to build-time errors. This helps to avoid obvious errors and to create cleaner APIs for our functions. You can find a nice cheat sheet here.

git

Effective communication makes software scale. Git is our medium and a very effective one to communicate our changes. Your changes should tell a story that’s easy to follow, so that the code can easily be picked up by others by a mere examination of its commit and list history. There needs to be structure to the changes that are sent, so that they can be easily reviewed, and held to high standards.

Commit message conventions

Commit message format Each commit message consists of a header, a body, and a footer.

<header>
<BLANK LINE>
<body>

Commit message header format

<type>: <short summary>
   |         |
   |         └─ Imperative, present tense. Not capitalized.
   |            No period at the end. Less than 50 characters.
   |
   └─ Commit Type: ci|docs|feat|fix|refactor|test

  • ci: Changes to our CI configuration files and scripts (example scopes: Jenkins)

  • docs: Documentation only changes

  • feat: A new feature

  • fix: A bug fix

  • refactor: A code change that neither fixes a bug nor adds a feature

  • test: Adding missing tests or correcting existing tests

Commit message body format

  • Use the imperative, present tense (e.g. ‘fix’ not ‘fixed’, ‘fixes’).

  • Wrap at 72 characters.

  • Use the body to explain what and why vs. how. Explain the motivation for the change.

  • You can include a comparison of the previous behavior with the new behavior in order to illustrate the impact of the change.

  • You can reference issues with #SENSYS-<issue number>.

Commit message template

You can install our .gitmessage to have a pre-filled template within your git commits.

git config commit.template .gitmessage

Branching conventions

We use the git-flow branching model. Please find a detailed description of the branching model in Bitbucket Tutorial and Vincent Driessen’s Blogpost.

Branch name format

<type>/SENSYS-<issue-number>-<short description>
   |                |               |
   |                |               |
   |                |               └─ Short, crisp description. Hyphen as separator between words.
   |                |
   |                └─ Issue number referencing the issue.
   |
   └─ Commit Type: bugfix|feature|hotfix|release

Tools

All used tools are open-source and have a pretty good online documentation. So, if you get stuck at some point, there is a high chance that somebody else had already the same problem and you find a solution in the documentation or on stackoverflow :)

Automation - Invoke

invoke is a Python (2.7 and 3.4+) task execution tool & library. We use it to automate our development infrastructure. invoke provides shortcuts with default parameters that are adequate for most situations, so you don’t have to think a lot about commandline arguments. You can list all available shortcuts with:

# List available options:
inv -l

# Run a specific command, e.g. build documentation
inv docs

The listed options will be explained in the following sections.

Documentation - sphinx

The documentation generation is built on sphinx. If you want to build the documentation locally, run:

inv docs

The built documentation can then be found in build/docs. The documentation of the current release lives at Jenkins.

Formatting - black, isort

Our code is automatically formatted with black and isort. There is no need to run the formatters manually, since they will be executed by our pre-commit hooks. If you want to do it nevertheless, here are the commands.

# Run black (-c to check the files only)
inv black

# Run isort (-c to check the files only)
inv isort

Static code analysis - mypy, pylint, flake8, pydocstyle

We perform static code analysis to detect simple errors, force high quality code and ensure some common codestyle. You can invoke the complete static analysis with:

inv static

In the following subsections, we will explain the tools used in the static code analysis and show how to invoke them separately.

mypy

You can use mypy to perform a static type checking of your code. This supports you to find incompatible function calls and function arguments without the need to actually run the code:

inv mypy

pylint

pylint is used to check your code for programming errors, to enforce a proper code style, helps to enforce a coding standard, sniffs for code smells and offers simple refactoring suggestions. When you run pylint, the code base will be rated with 10.0 as best score and 0.0 as worst score. It is invoked by:

inv pylint

flake8

flake8 is a tool to check your Python code against some of the style conventions in PEP 8. We use it to guarantee some common code style among all developers. You can perform the check with:

inv flake8

pydocstyle

pydocstyle is a static analysis tool for checking compliance with Python docstring conventions. We use it to guarantee some common doc style among all developers. You can perform the check with:

inv pydocstyle

Testing - pytest & coverage

We use pytest & coverage for running the test suite. pytest is used for the actual testing, while we use coverage to generate reports of our test coverage. Here is a short overview of important commands to run tests (find more information and more examples in the pytest documentation):

# Run all tests
pytest tests
# Run only a specific test
pytest tests -k name_of_the_test
# Run only a subset of tests, located in a specific folder
pytest tests/test_preprocessing

If you simply want to run all the tests & create a coverage report run:

# All tests
inv test
# Only unittests
inv unittest
# Only system tests
inv systest

Test and coverage reports will be available under build/test, build/unit and build/system.

Testing against multiple python versions - tox

tox is a tool for checking that your package installs correctly with different Python versions and interpreters. Besides that we can use it for running your tests in each of the environments. This supports us to test our package with various python versions. We can run it with

inv tox
